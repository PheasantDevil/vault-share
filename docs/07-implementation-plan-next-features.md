# 次の実装機能の設計計画

## 実装対象機能

1. **MFA（多要素認証）の実装（R2）**
2. **パスワードリセット機能の実装（R3の一部）**
3. **1Password 連携（CSV インポート/エクスポート）（R4の一部）**
4. **アイテム検索機能の追加（R10の拡張）**
5. **Cloud Run デプロイの自動化**

**注意**: レート制限・セキュリティヘッダ（R12）は実装不要のため、このドキュメントから除外しています。

---

## 1. MFA（多要素認証）の実装（R2）

### 1.1 要件の整理

| 項目               | 内容                                                                                                                             |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| **TOTP MFA**       | Identity Platform の TOTP（Time-based One-Time Password）を利用。Google Authenticator 等のアプリで QR コードをスキャンして登録。 |
| **SMS MFA**        | Identity Platform の SMS を利用。1日10通まで無料。                                                                               |
| **MFA 設定**       | ユーザーが MFA を有効化/無効化できる設定画面を提供。                                                                             |
| **ログインフロー** | ログイン時に MFA が有効な場合は、MFA コード入力画面を表示。                                                                      |

### 1.2 コンポーネント設計

| コンポーネント/層           | 役割                                     | Props / State 等                                                         |
| --------------------------- | ---------------------------------------- | ------------------------------------------------------------------------ |
| **MFA 設定ページ**          | MFA の有効化/無効化、TOTP 登録、SMS 登録 | MFA 状態、QR コード、エラーメッセージ                                    |
| **MFA 登録 API**            | TOTP/SMS の登録処理                      | 入力: MFA タイプ（TOTP/SMS）、出力: QR コード（TOTP）または SMS 送信結果 |
| **MFA 検証 API**            | 登録時の検証コード確認                   | 入力: コード、出力: 成功/失敗                                            |
| **MFA ログイン検証 API**    | ログイン時の MFA コード検証              | 入力: ID トークン、MFA コード、出力: 検証済みトークン                    |
| **MFA ログイン画面**        | ログイン後の MFA コード入力画面          | フォーム state、エラーメッセージ                                         |
| **セッション発行 API 拡張** | MFA 検証済みトークンでセッション発行     | 既存のセッション発行 API を拡張                                          |

### 1.3 データ設計・状態管理

- **Firestore**: `users` コレクションに `mfaEnabled` フィールドを追加（既存の `UserDoc` に含まれる）
- **Identity Platform**: MFA の登録・検証は Identity Platform の API を使用
- **セッション**: MFA 検証済みの ID トークンでセッションを発行

### 1.4 実装順序

1. **Identity Platform で MFA を有効化**（コンソール設定）
2. **MFA 設定ページの実装**（`/dashboard/settings`）
3. **MFA 登録 API の実装**（TOTP/SMS）
4. **MFA 検証 API の実装**（登録時・ログイン時）
5. **ログインフローの拡張**（MFA チェック → コード入力画面）
6. **セッション発行 API の拡張**（MFA 検証済みトークンの処理）

### 1.5 懸念事項・エッジケース

| 懸念                          | 対策                                                                                         |
| ----------------------------- | -------------------------------------------------------------------------------------------- |
| **SMS の無料枠超過**          | 1日10通まで無料。超過時は TOTP を推奨するメッセージを表示。                                  |
| **MFA デバイスの紛失**        | 管理者による MFA 無効化機能を検討（将来対応）。現時点は手動で Identity Platform から無効化。 |
| **ログイン時の MFA チェック** | ログイン成功後に MFA 状態を確認し、必要に応じてコード入力画面へリダイレクト。                |

## 2. パスワードリセット機能の実装（R3の一部）

### 2.1 要件の整理

| 項目                             | 内容                                             |
| -------------------------------- | ------------------------------------------------ |
| **パスワードリセットリクエスト** | メールアドレスを入力してリセットメールを送信。   |
| **パスワードリセット完了**       | メールリンクから遷移し、新しいパスワードを設定。 |

### 2.2 コンポーネント設計

| コンポーネント/層                      | 役割                                                     | Props / State 等                                          |
| -------------------------------------- | -------------------------------------------------------- | --------------------------------------------------------- |
| **パスワードリセットリクエストページ** | メールアドレス入力フォーム                               | フォーム state、エラーメッセージ、送信完了フラグ          |
| **パスワードリセット完了ページ**       | 新しいパスワード設定フォーム                             | フォーム state、エラーメッセージ、リセットトークン        |
| **パスワードリセットリクエスト API**   | Identity Platform の `sendPasswordResetEmail` を呼び出し | 入力: メールアドレス、出力: 成功/失敗                     |
| **パスワードリセット完了 API**         | Identity Platform の `confirmPasswordReset` を呼び出し   | 入力: リセットトークン、新しいパスワード、出力: 成功/失敗 |

### 2.3 データ設計・状態管理

- **Identity Platform**: パスワードリセットのトークン管理は Identity Platform が行う
- **Firestore**: 追加のデータ保存は不要

### 2.4 実装順序

1. **パスワードリセットリクエストページの実装**（`/reset-password`）
2. **パスワードリセットリクエスト API の実装**
3. **パスワードリセット完了ページの実装**（`/reset-password/confirm`）
4. **パスワードリセット完了 API の実装**
5. **ログインページに「パスワードを忘れた場合」リンクを追加**

### 2.5 懸念事項・エッジケース

| 懸念                           | 対策                                                                                                             |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| **許可外メールでのリセット**   | 許可リストに含まれていないメールアドレスでもリセットメールは送信するが、実際のリセット時には許可チェックを行う。 |
| **リセットトークンの有効期限** | Identity Platform のデフォルト設定（通常1時間）を使用。                                                          |
| **レート制限**                 | Identity Platform 側でレート制限がかかる。必要に応じて追加のレート制限を実装（R12で対応）。                      |

## 3. 1Password 連携（CSV インポート/エクスポート）（R4の一部）

### 3.1 要件の整理

| 項目                 | 内容                                                              |
| -------------------- | ----------------------------------------------------------------- |
| **CSV インポート**   | 1Password の CSV エクスポート形式をパースし、アイテムとして登録。 |
| **CSV エクスポート** | 当サービスのアイテムを 1Password CSV 形式でエクスポート。         |

### 3.2 コンポーネント設計

| コンポーネント/層        | 役割                                           | Props / State 等                                           |
| ------------------------ | ---------------------------------------------- | ---------------------------------------------------------- |
| **CSV インポートページ** | ファイルアップロード、プレビュー、グループ選択 | ファイル、プレビューデータ、選択グループ、エラーメッセージ |
| **CSV インポート API**   | CSV パース、アイテム登録                       | 入力: ファイル、グループID、出力: 登録結果                 |
| **CSV エクスポート API** | アイテム取得、CSV 生成                         | 入力: グループID、出力: CSV ファイル                       |
| **CSV エクスポート UI**  | グループ選択、ダウンロード                     | 選択グループ、ダウンロード状態                             |
| **CSV パーサー**         | 1Password CSV 形式の解析                       | 入力: CSV 文字列、出力: パース済みアイテム配列             |

### 3.3 データ設計・状態管理

- **CSV 形式**: 1Password の標準 CSV エクスポート形式（`Title`, `Website`, `Username`, `Password`, `Notes`, `Type` 等）
- **マッピング**: CSV の列を当サービスの `ItemPayload` にマッピング
- **エクスポート**: `ItemPayload` を 1Password CSV 形式に変換

### 3.4 実装順序

1. **CSV パーサーの実装**（`packages/csv-parser/` または `apps/web/src/lib/csv/`）
2. **CSV インポート API の実装**（`/api/groups/[id]/import`）
3. **CSV インポート UI の実装**（グループ詳細ページに追加）
4. **CSV エクスポート API の実装**（`/api/groups/[id]/export`）
5. **CSV エクスポート UI の実装**（グループ詳細ページに追加）

### 3.5 懸念事項・エッジケース

| 懸念                             | 対策                                                                                |
| -------------------------------- | ----------------------------------------------------------------------------------- |
| **大量データの処理**             | チャンク処理またはバックグラウンドジョブで実装。進捗をポーリングまたは SSE で返す。 |
| **CSV 形式のバリエーション**     | 1Password の標準形式を優先し、他の形式はエラーメッセージで対応。                    |
| **ファイルサイズ制限**           | 10MB 程度の制限を設ける。                                                           |
| **エクスポート時の暗号化データ** | 復号してから CSV に出力。                                                           |

## 4. アイテム検索機能の追加（R10の拡張）

### 4.1 要件の整理

| 項目                       | 内容                                               |
| -------------------------- | -------------------------------------------------- |
| **タイトル検索**           | アイテム一覧でタイトルを検索できるようにする。     |
| **フィルタとの組み合わせ** | 既存の type フィルタと組み合わせて検索可能にする。 |

### 4.2 コンポーネント設計

| コンポーネント/層 | 役割                           | Props / State 等                         |
| ----------------- | ------------------------------ | ---------------------------------------- |
| **検索ボックス**  | タイトル検索の入力欄           | 検索クエリ、検索状態                     |
| **検索ロジック**  | クライアント側でタイトルを検索 | 検索クエリ、アイテムリスト、フィルタ状態 |

### 4.3 データ設計・状態管理

- **クライアント側検索**: 既に取得済みのアイテムリストをクライアント側でフィルタリング
- **サーバー側検索**: 将来的に必要になった場合は、Firestore の全文検索や Algolia 等を検討

### 4.4 実装順序

1. **グループ詳細ページに検索ボックスを追加**
2. **検索ロジックの実装**（クライアント側でタイトル検索）
3. **フィルタとの組み合わせ**（type フィルタと検索クエリの両方を適用）

### 4.5 懸念事項・エッジケース

| 懸念                             | 対策                                                                 |
| -------------------------------- | -------------------------------------------------------------------- |
| **暗号化データの検索**           | タイトルは復号後に検索するため、一覧取得時に復号済みタイトルを返す。 |
| **大量データ時のパフォーマンス** | クライアント側検索は数百件程度まで。それ以上はサーバー側検索を検討。 |

## 5. Cloud Run デプロイの自動化

### 5.1 要件の整理

| 項目               | 内容                                                                            |
| ------------------ | ------------------------------------------------------------------------------- |
| **自動デプロイ**   | main ブランチへのマージ時に Cloud Run に自動デプロイ。                          |
| **プレビュー環境** | PR 作成時にプレビュー用リビジョンをデプロイ（トラフィック 0%）。                |
| **OIDC 認証**      | GitHub Actions から GCP への認証は OIDC（Workload Identity Federation）を使用。 |

### 5.2 コンポーネント設計

| コンポーネント/層        | 役割                                     | 入出力                                    |
| ------------------------ | ---------------------------------------- | ----------------------------------------- |
| **デプロイワークフロー** | GitHub Actions で Cloud Run にデプロイ   | トリガー: main への push、PR の push      |
| **コンテナビルド**       | Next.js アプリを Docker イメージにビルド | 入力: ソースコード、出力: Docker イメージ |
| **Artifact Registry**    | Docker イメージを保存                    | 入力: Docker イメージ、出力: イメージ URI |
| **Cloud Run デプロイ**   | Cloud Run サービスにデプロイ             | 入力: イメージ URI、出力: サービス URL    |

### 5.3 データ設計・状態管理

- **GitHub Secrets**: OIDC 認証に必要な情報は GitHub リポジトリ変数に設定（既に設定済み）
- **環境変数**: Cloud Run の環境変数は初回デプロイ後に手動設定（`docs/05-deployment/03-cicd-setup-gcp-and-github.md` 参照）

### 5.4 実装順序

1. **Dockerfile の確認・修正**（既存の `Dockerfile` を確認）
2. **デプロイワークフローの実装**（`.github/workflows/deploy.yml`）
3. **プレビュー環境の実装**（PR 用のデプロイ）
4. **動作確認**（main へのマージでデプロイが実行されることを確認）

### 5.5 懸念事項・エッジケース

| 懸念                             | 対策                                                                                                      |
| -------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **環境変数の管理**               | 初回デプロイ後に Cloud Run の環境変数を手動設定。Secret Manager を使用する場合は `--set-secrets` を使用。 |
| **Firebase の承認済みドメイン**  | Cloud Run の URL を Firebase の承認済みドメインに追加する必要がある。                                     |
| **デプロイ失敗時のロールバック** | Cloud Run のリビジョン管理で自動ロールバック可能。                                                        |

## 実装順序の推奨

1. **パスワードリセット機能**（比較的シンプルで、MFA の前提となる）
2. **MFA の実装**（認証機能の完成）
3. **アイテム検索機能**（UI の改善、比較的簡単）
4. **1Password CSV 連携**（データ移行機能）
5. **Cloud Run デプロイ自動化**（運用効率化）

**注意**: レート制限・セキュリティヘッダ（R12）は実装不要のため、実装順序から除外しています。
