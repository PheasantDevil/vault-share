# 機密情報共有 Web サービス — アーキテクチャ・要件設計

## 1. 要件一覧

### 1.1 必須要件（初期リリース）

| ID  | 要件                        | 備考                                                                                                                                                                                                                                                                             |
| --- | --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | **認証管理**                | Google Cloud Identity Platform（0–49,999 MAU 無料）。メール/パスワード + ソーシャル（Google 等）。                                                                                                                                                                               |
| R2  | **MFA（2FA）**              | Identity Platform の無料枠内で TOTP および SMS（1日10通まで無料）をサポート。                                                                                                                                                                                                    |
| R3  | **ログイン機能**            | セッション管理・ログアウト・パスワードリセット。                                                                                                                                                                                                                                 |
| R4  | **データ連携（1Password）** | **初期リリースに 1Password Connect を含める**（無料枠 3 vault を利用）。vault 参照・アイテム選択して当サービスに登録可能。**無料枠超過時は CSV インポートに切り替える**（1Password エクスポートのインポートおよび当サービスから 1Password 形式へのエクスポートは常に利用可能）。 |
| R5  | **データ登録**              | 手動登録（タイトル・URL・ユーザー名・パスワード・メモ等）。将来の Chrome 拡張・1Password 連携を見据えた API/データ形式を用意。                                                                                                                                                   |
| R6  | **グループ間共有**          | 少人数チームを想定した「グループ」を作成し、グループ単位で機密項目を共有する。                                                                                                                                                                                                   |
| R7  | **暗号化**                  | 機密項目の本体はアプリケーション層で AES-256 暗号化して保存。鍵は GCP Secret Manager。転送は TLS。                                                                                                                                                                               |
| R8  | **監査・アクセス制御**      | 認証済みユーザーのみが自分が属するグループのデータにアクセス。参照・作成・更新・削除の監査ログ。                                                                                                                                                                                 |

### 1.2 初期段階で「最低限あるとよい」追加提案

| ID  | 要件                               | 理由                                                                                    |
| --- | ---------------------------------- | --------------------------------------------------------------------------------------- |
| R9  | **招待・メンバー管理**             | グループへの招待（メールリンク等）、メンバーの追加/削除。共有サービスとして必須に近い。 |
| R10 | **項目のカテゴリ/ラベル**          | ログイン・銀行・APIキー等の種別やラベルでフィルタ・検索しやすくする。                   |
| R11 | **論理削除**                       | 誤削除対策と監査のため、削除は論理削除とし、一定期間後に物理削除するオプション。        |
| R12 | **レート制限・セキュリティヘッダ** | ブルートフォース・不正利用対策。GCP や Next.js ミドルウェアで実装。                     |

### 1.3 スコープ外（将来対応）

- **Chrome 拡張**: 初期は「1Password エクスポートのインポート」で対応。拡張の開発は機能強化時に行い、README に方針を記載する。
- **E2E クライアント側暗号化**: サーバーを介さずクライアントのみで復号する方式は、検索・共有の複雑さのため初期では対象外。

## 2. コンポーネント設計

### 2.1 システム構成（論理）

```
[ユーザー]
    ↓ HTTPS
[Next.js App] ←→ [Identity Platform]
    ↓               (認証・MFA)
[API Routes / Server Actions]
    ↓
[バックエンド API 層]（Next.js API または BFF）
    ↓
[DB] + [Secret Manager]（暗号化鍵）
[1Password Connect]（初期リリースに含む。超過時は CSV インポートに切替）
```

### 2.2 フロントエンド（Next.js 想定）

| コンポーネント/領域         | 役割                                              | 主な Props/State                                       |
| --------------------------- | ------------------------------------------------- | ------------------------------------------------------ |
| **Layout / AuthGuard**      | 認証状態に応じたリダイレクト、共通レイアウト      | セッション、子要素                                     |
| **Login / SignUp / MFA**    | ログイン・登録・MFA 入力                          | フォーム state、エラー、Identity Platform コールバック |
| **Dashboard**               | グループ一覧・直近の項目一覧                      | グループ一覧、項目サマリ                               |
| **GroupList / GroupDetail** | グループの一覧・詳細・メンバー・招待              | グループ ID、メンバー、招待リンク                      |
| **ItemList / ItemDetail**   | 機密項目の一覧・詳細・作成・編集                  | 項目リスト、選択項目、フィルタ（カテゴリ等）           |
| **ItemForm**                | 手動登録・編集フォーム                            | 初期値、送信コールバック、バリデーション               |
| **1Password Import/Export** | ファイルアップロード（CSV/1PUX）、エクスポート UI | ファイル、マッピング設定、エクスポート形式             |
| **Settings**                | プロフィール・MFA 設定・パスワード変更            | ユーザー設定、MFA 状態                                 |

- **状態管理**: サーバー状態は React Query（TanStack Query）等でキャッシュ。グローバルな認証状態は Context または Identity Platform のクライアント SDK とセッションで保持。

### 2.3 バックエンド（API・ドメイン）

| コンポーネント          | 役割                                                                                          | 入出力                                    |
| ----------------------- | --------------------------------------------------------------------------------------------- | ----------------------------------------- |
| **Auth サービス**       | Identity Platform との連携、セッション検証、MFA フロー                                        | IdP トークン、セッション Cookie           |
| **User サービス**       | ユーザー情報・プロフィールの取得・更新                                                        | UID、表示名、メール                       |
| **Group サービス**      | グループ CRUD、メンバー追加/削除、招待                                                        | グループ ID、メンバーリスト、招待トークン |
| **Item サービス**       | 機密項目の CRUD、リスト、検索（暗号化/復号を含む）                                            | 項目 DTO、グループ ID、ユーザー権限       |
| **1Password 連携**      | インポート（CSV/1PUX パース）、エクスポート、Connect API（初期より利用。超過時は CSV に切替） | ファイル or Connect レスポンス、項目 DTO  |
| **Encryption サービス** | 項目の暗号化/復号、鍵は Secret Manager から取得                                               | 平文/暗号文、鍵 ID                        |

### 2.4 インフラ（GCP）

- **IaC**: Pulumi（TypeScript）。リポジトリ内 `infra/` に配置。
- **想定リソース**: Cloud Run（Next.js または API サーバー）、**Firestore**（データストア）、Secret Manager、Identity Platform、Artifact Registry、必要に応じて VPC・ロードバランサ。
- **CI/CD**: GitHub Actions。main ブランチへのマージまたはタグでビルド・テスト・デプロイ。

## 3. データ設計・状態管理

### 3.1 主要エンティティ

- **User**: Identity Platform の UID をキーに、表示名・メール・MFA 状態等を保持（必要なら自前テーブル/コレクションで拡張）。
- **Group**: グループ ID、名前、作成者、作成日時。メンバーは **GroupMember** で管理。
- **GroupMember**: groupId、userId、role（owner/member）、参加日時。
- **Invitation**: グループ招待。token、groupId、招待者、有効期限、使用済みフラグ。
- **Item（機密項目）**: id、groupId、タイトル・URL・ユーザー名・パスワード・メモ・カテゴリ等。**保存時はこれらをまとめて AES で暗号化**し、DB には ciphertext + iv 等を保存。検索用にグループ ID・作成者・作成日・論理削除フラグは平文で保持可能。

### 3.2 状態管理方針

- **サーバー**: **Firestore** を採用。少人数・スキーマ変更のしやすさと無料枠を考慮。無料枠を超えそうな場合のコスト目安は README に記載する。
- **フロント**: 認証は Identity Platform のセッション。API データは TanStack Query でキャッシュし、ミューテーション後に invalidate。フォームはローカル state または React Hook Form 等。

## 4. 懸念事項・エッジケース・パフォーマンス

| 懸念                             | 対策                                                                                                                                          |
| -------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **鍵の漏洩・ローテーション**     | 鍵は Secret Manager のみに保持。ローテーション時は新鍵で再暗号化するバッチまたは手順を設計に含める。                                          |
| **1Password Connect 無料枠超過** | 3 vault まで無料。超過時は **CSV インポート**に切り替える。README に利用規約・切替方法を明記。                                                |
| **インポート時の重いリクエスト** | 大きな CSV/1PUX はチャンク処理またはバックグラウンドジョブで取り込み、進捗をポーリングまたは SSE で返す。                                     |
| **招待トークンの悪用**           | トークンは推測困難にし、有効期限と 1 回限り使用。レート制限をかける。                                                                         |
| **監査ログの保持期間**           | GCP のログ保持ポリシーと合わせ、設計書に「何日/何年保持するか」を記載。                                                                       |
| **コスト**                       | GCP 無料枠（Cloud Run・Firestore・Secret Manager）内で収まるよう、リソース上限を IaC で明示。Firestore の無料枠超過時の目安は README に記載。 |
| **多言語・タイムゾーン**         | 海外ユーザー想定のため、日付・時刻は UTC 保存・表示でロケール対応の余地を残す。                                                               |

## 5. 命名・リポジトリ構成

### 5.1 リポジトリ名案

- **推奨**: `vault-share`
  - 理由: 短く、機密情報の「保管・共有」を連想しやすく、英語でそのまま説明できる。
- 代替: `safe-share`, `creds-vault` 等。

### 5.2 リポジトリ構成（モノレポ案）

```
/Users/Work/vault-share/
├── README.md                 # プロジェクト概要・Chrome 拡張の将来実装について記載
├── docs/                     # 本設計・調査メモ（01-research, 02-design）
├── apps/
│   ├── web/                  # Next.js フロント + API Routes / Server Actions
│   └── (extension)/          # 将来: Chrome 拡張（現時点はディレクトリまたは README のみ）
├── packages/
│   ├── api-client/           # フロント用 API クライアント・型
│   ├── db/                   # DB スキーマ・マイグレーション・クライアント（共有）
│   └── crypto/               # 暗号化/復号の共通ロジック（鍵取得はアプリ層）
├── infra/                    # Pulumi（TypeScript）で GCP を定義
├── .github/
│   └── workflows/            # GitHub Actions（lint, test, deploy）
├── package.json              # ルートのワークスペース（npm/pnpm/yarn workspaces）
└── ...
```

- **モノレポを採用する理由**: 調査メモ `03-monorepo-vs-multi-repo.md` に記載。少人数・型共有・原子的な変更を優先。

### 5.3 CI/CD（GitHub Actions）方針

- **コスト**: GitHub Actions の無料枠内で実施。GCP へのデプロイは OIDC で認証し、キーをリポジトリに置かない。
- **ワークフロー案**:
  - `main` への push / PR: ESLint, Prettier, TypeScript チェック、ユニットテスト（Vitest 等）。
  - `main` マージ時: ビルド、E2E（Playwright 等はオプション）、Pulumi によるインフラ更新、Cloud Run 等へのデプロイ。
- **環境**: 開発（dev）、本番（prod）を分離。ブランチまたはタグでデプロイ先を切り替え。
