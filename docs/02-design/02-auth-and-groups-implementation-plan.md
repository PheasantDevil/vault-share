# 認証（A）とグループ・招待（B）の実装計画

## 方針

- **A（認証）を先に実施**し、そのあと **B（グループ・招待）** を実施する。
- B のうち **A と同時にやる方がよい作業** は A に含める（下記「A に含める B 関連」）。

---

## 1. 要件の整理

### 1.1 A: 認証まわりで実装するもの

| 項目                 | 内容                                                                                                                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **ログイン**         | Identity Platform のメール/パスワードのみ。クライアントで `signInWithEmailAndPassword`、サーバーで ID トークン検証。                                                                 |
| **許可メールのみ**   | ログイン・新規登録とも、**許可リストに載っているメールアドレスのみ**成功とする。許可外の場合は「このメールアドレスは登録されていません」等のメッセージで拒否。                       |
| **許可リストの保持** | **環境変数 `ALLOWED_EMAILS`**（カンマ区切り）で保持する。運用で変更しやすく、Firestore 読み取りを増やさない。本番で人数が増える場合はのちに Firestore コレクションへ移行可能とする。 |
| **新規登録**         | 許可メールのみ受け付ける。許可リストにないメールでのサインアップは「このメールアドレスは招待されていません」等で拒否。                                                               |
| **セッション**       | サーバーで ID トークンを検証し、検証成功時に **HTTP-only のセッション Cookie**（または Next.js の Session）を発行。以降のリクエストは Cookie で認証済みユーザーを識別。              |
| **ログアウト**       | クライアントで IdP の signOut、サーバーでセッション Cookie を削除。                                                                                                                  |
| **保護ルート**       | 未ログインで `/dashboard` 等にアクセスした場合は `/login` へリダイレクト。                                                                                                           |

### 1.2 A に含める B 関連（A と同時に実施）

| 項目                                             | 理由                                                                                                                                                                                                                        |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ログイン成功時に Firestore に User 作成/更新** | B でグループ・メンバーを扱うとき、メンバーは `userId` で User を参照する。A の段階で「ログインするたびに users にドキュメントが存在する」ようにしておくと、B でグループ CRUD を実装する際にユーザー作成を考慮しなくてよい。 |
| **User ドキュメントの形**                        | 既存の `packages/db` の `UserDoc`（uid, email, displayName, createdAt, updatedAt）をそのまま利用。初回ログイン時は作成、2 回目以降は updatedAt 等を更新。                                                                   |

### 1.3 B: グループ・招待で実装するもの（A 完了後に実施）

| 項目           | 内容                                                                                                                                      |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **Group CRUD** | グループの作成・一覧・詳細・更新・削除。Firestore の `groups` / `groupMembers` を使用（スキーマは既存の `GroupDoc` / `GroupMemberDoc`）。 |
| **招待**       | 招待トークン発行・招待リンク・トークン検証による参加。`InvitationDoc` を利用。                                                            |

## 2. コンポーネント設計

### 2.1 A: 認証まわり

| コンポーネント/層                      | 役割                                                                                                                                                                                                                                                                                                                          | Props / State 等                          |
| -------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| **Firebase クライアント初期化**        | `apps/web` で Firebase JS SDK を初期化（API Key 等は環境変数）。ブラウザのみで利用。                                                                                                                                                                                                                                          | -                                         |
| **Login ページ**                       | メール・パスワード入力フォーム。送信で `signInWithEmailAndPassword`。成功後はサーバーに ID トークンを送り、セッション発行 → リダイレクト。許可外メールの場合は IdP 前にチェック（またはサインイン失敗時にサーバー側で許可チェックしてメッセージ返却）。                                                                       | フォーム state、エラーメッセージ、loading |
| **SignUp ページ**                      | メール・パスワード・表示名入力。送信前に **許可リストチェック**（サーバーに「このメールは許可されているか」を問い合わせる API を用意するか、createUser 後にサーバーでチェックして不許可ならユーザー削除＋エラー）。推奨: サインアップ成功後にサーバーでトークン検証＋許可チェックし、不許可ならセッション発行せずエラー返却。 | 同上                                      |
| **セッション発行 API / Server Action** | クライアントから ID トークン（またはカスタムトークン）を受け取り、firebase-admin で検証。**許可リストに含まれるか**確認。OK なら Firestore に User を作成/更新し、セッション Cookie をセットして 200。NG なら 403 とメッセージ。                                                                                              | 入力: idToken。出力: 成功/失敗            |
| **許可チェック**                       | サーバー側で `ALLOWED_EMAILS` を読み、ログイン/登録しようとしているメールが含まれるか判定。                                                                                                                                                                                                                                   | 入力: email。出力: boolean                |
| **AuthGuard / ミドルウェア**           | 特定パス（例: `/dashboard` 以降）ではセッションを確認。無ければ `/login` へリダイレクト。                                                                                                                                                                                                                                     | -                                         |
| **ログアウト**                         | クライアントで IdP signOut、サーバーでセッション Cookie 削除（API または Server Action）。                                                                                                                                                                                                                                    | -                                         |
| **User 作成/更新**                     | セッション発行時、IdP の UID・メール・表示名で `users/{uid}` を set（存在しなければ作成、あれば updatedAt 等を更新）。`@vault-share/db` の getDb() と COLLECTIONS.users を使用。                                                                                                                                              | -                                         |

### 2.2 B: グループ・招待（参照のみ、実装は B フェーズで）

| コンポーネント          | 役割                                   |
| ----------------------- | -------------------------------------- |
| GroupList / GroupDetail | グループ一覧・詳細・作成・更新・削除   |
| 招待 API / UI           | トークン発行・招待リンク・トークン検証 |

## 3. データ設計・状態管理

### 3.1 認証・セッション

- **クライアント**: Firebase Auth の状態（onAuthStateChanged）で「ログイン中か」を判定。ログイン成功後、ID トークンをサーバーに送ってセッション発行。
- **サーバー**: セッションは **Next.js の Cookie**（または `jose` 等で署名した JWT を Cookie に載せる）で保持。リクエストごとに Cookie を検証し、uid を取得。
- **許可リスト**: 環境変数 `ALLOWED_EMAILS`（例: `a@example.com,b@example.com`）。サーバーのみから参照。

### 3.2 Firestore（A で触れる部分）

- **users**: ログイン成功時に `users/{uid}` を set。`UserDoc` の形で createdAt / updatedAt を更新。
- **groups / groupMembers / invitations**: A では読み書きしない。B で使用。

### 3.3 環境変数（A で追加）

| 変数                                   | 説明                                                                                              | 例                                    |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------- |
| `NEXT_PUBLIC_FIREBASE_API_KEY`         | クライアント用 Firebase API Key                                                                   | （Firebase コンソールで取得）         |
| `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`     | Auth ドメイン                                                                                     | `vault-share-dev.firebaseapp.com`     |
| `NEXT_PUBLIC_FIREBASE_PROJECT_ID`      | プロジェクト ID                                                                                   | `vault-share-dev`                     |
| `ALLOWED_EMAILS`                       | 許可するメールアドレス（カンマ区切り）                                                            | `user1@example.com,user2@example.com` |
| `SESSION_SECRET`                       | セッション JWT 署名用（16 文字以上推奨）                                                          | ランダム文字列                        |
| （サーバーで ID トークン検証する場合） | Firebase Admin の認証情報は GCP のデフォルト（ADC）または `GOOGLE_APPLICATION_CREDENTIALS` で取得 | -                                     |

**運用**: `apps/web/env.example` を `.env.local` にコピーして値を設定する。

## 4. 懸念・エッジケース

| 懸念                                                              | 対策                                                                                                                                               |
| ----------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 許可外メールで IdP にサインアップ済みのユーザーがログインしてくる | サーバーでトークン検証後に必ず許可リストをチェック。許可外ならセッション発行せず 403。「このメールアドレスは利用できません」等のメッセージを返す。 |
| トークン期限切れ                                                  | クライアントで refresh するか、セッション Cookie の有効期限を適切に設定。期限切れ時は再ログインへ誘導。                                            |
| ALLOWED_EMAILS の前後の空白                                       | サーバーで split した各要素を trim してから比較する。                                                                                              |
| 本番で人数が増え環境変数が長くなる                                | のちに Firestore の `config/allowedEmails` のようなドキュメントやコレクションに移行する選択肢を残す。                                              |

## 5. 実装順序（A フェーズ）

1. **環境変数と Firebase クライアント初期化**
   - `.env.example` に上記変数を記載。
   - `apps/web` に Firebase 初期化モジュール（クライアント用）を追加。

2. **許可リストチェック API**
   - サーバー側で `ALLOWED_EMAILS` を読んでチェックする関数を用意（セッション発行時に使用）。

3. **セッション発行 API（または Server Action）**
   - 入力: ID トークン。
   - 処理: firebase-admin でトークン検証 → メールを許可リストと照合 → User 作成/更新 → セッション Cookie セット。
   - 失敗時: 403 とメッセージ。

4. **ログイン画面**
   - メール/パスワードフォーム → `signInWithEmailAndPassword` → 成功後 ID トークンをセッション発行 API に送信 → 成功したら `/dashboard`（またはトップ）へリダイレクト。許可外の場合は API が 403 を返すのでそのメッセージを表示。

5. **新規登録画面**
   - メール/パスワード/表示名フォーム → `createUserWithEmailAndPassword` → 同上でセッション発行 API に送信。許可外なら 403 のメッセージを表示。

6. **ログアウト**
   - クライアントで signOut、サーバーでセッション Cookie 削除する API を呼び、`/login` へリダイレクト。

7. **AuthGuard / ミドルウェア**
   - 保護したいパスではセッションを確認し、無ければ `/login` へリダイレクト。ダッシュボード用のプレースホルダーページを 1 つ用意してもよい。

8. **User 作成/更新**
   - セッション発行時に `users/{uid}` を Firestore に set。`packages/db` の getDb() と UserDoc 型を使用。

**※ A フェーズは上記のとおり実装済み**（Firebase クライアント/Admin、許可リスト、セッション API、ログイン/サインアップ/ログアウト、ミドルウェアで `/dashboard` 保護、ログイン時の User 作成/更新）。

---

## 6. 実装順序（B フェーズ・A 完了後）

1. Group CRUD の API と Firestore 書き込み。
2. グループ一覧・詳細・作成・編集・削除の画面。
3. 招待トークン発行・招待リンク・トークン検証による参加処理。

**※ B フェーズは上記のとおり実装済み**（Group CRUD API、メンバー一覧 API、招待発行 API、招待参加 API、ダッシュボード・グループ一覧・作成・詳細・編集・削除画面、招待ページ）。

---

この計画で問題なければ、「この設計で実装に進めてよい」と指示いただければ、A フェーズから実装を進めます。
