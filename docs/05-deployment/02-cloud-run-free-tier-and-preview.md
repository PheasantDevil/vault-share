# Cloud Run 無料枠とプレビュー URL 方針

本サービスはインフラを GCP に統一するため、フロントエンド（Next.js）のデプロイ先を **Cloud Run** とする。

---

## 1. Cloud Run 無料枠の整理

### 1.1 無料枠のリセット

- **毎月リセット**される。
- 無料枠は**同一課金アカウントに紐づく全プロジェクトで合算**される（プロジェクトごとではない）。

### 1.2 無料枠の内容（リクエストベース課金＝デフォルト）

| リソース         | 毎月の無料枠    | 備考            |
| ---------------- | --------------- | --------------- |
| **リクエスト数** | 200 万回        | 超過分から課金  |
| **vCPU 時間**    | 180,000 vCPU-秒 | 約 50 vCPU-時間 |
| **メモリ**       | 360,000 GiB-秒  | 約 100 GiB-時間 |

- **インスタンスベース課金**を選んだ場合: リクエストの無料枠はなく、vCPU 24万秒・GiB-秒 45万秒が無料。
- 本プロジェクトでは**リクエストベース（デフォルト）**を想定。

### 1.3 無料枠が適用されるリージョン

- **Tier 1 リージョン**で無料枠が適用される。
- 例: `us-central1`（アイオワ）、`asia-northeast1`（東京）、`asia-northeast2`（大阪）、`asia-east1`（台湾）など。
- 日本からは **`asia-northeast1`** を推奨（レイテンシ・無料枠の両立）。

### 1.4 どんな使い方で課金が発生するか

以下のいずれかを**月間で無料枠超過**すると、超過分から課金される。

| 要因             | 無料枠内の目安          | 超過すると                                                 |
| ---------------- | ----------------------- | ---------------------------------------------------------- |
| **リクエスト数** | 200 万回/月以内         | 超過分の単価（約 $0.40/100万リクエスト）で課金             |
| **vCPU 時間**    | 180,000 vCPU-秒/月以内  | 超過分の vCPU-秒単価で課金                                 |
| **メモリ**       | 360,000 GiB-秒/月以内   | 超過分の GiB-秒単価で課金                                  |
| **送信データ**   | 北米向け 1 GiB/月が無料 | それを超える送信は従量課金（リージョンにより単価は異なる） |

- 課金は**実際に消費したリソース**に対してのみ発生し、100 ミリ秒単位で端数切り上げ。
- 少人数・低トラフィックであれば、**毎月無料枠内に収まる運用**が現実的。

- [Cloud Run 料金](https://cloud.google.com/run/pricing)

## 2. プレビュー URL の代替案（Vercel のような「デプロイごとに URL」）

Cloud Run は Vercel のように「PR ごとに自動でプレビュー URL が発行される」機能は標準では持たない。以下の 3案で検討し、**現時点で最も有効な案で固定**する。

### 案 A: PR 時に Cloud Run へプレビュー用リビジョンをデプロイし、PR コメントに URL を投稿

- **内容**: GitHub Actions で PR の push 時に、Cloud Run に**新リビジョン**をデプロイ（トラフィック 0% のプレビュー用）。デプロイ後に**PR にコメント**でプレビュー URL を投稿する。
- **メリット**: PR ごとに確認用 URL が得られ、レビューしやすい。Vercel のプレビューに近い体験。
- **デメリット**: ワークフローと権限の設定が必要。プレビュー用リビジョンが増えるとリソース管理の考慮が必要（リビジョン数・古いリビジョンの削除など）。
- **実装**: 自前の GitHub Actions ワークフロー（`gcloud run deploy` でタグ付きリビジョン or 別サービス名）＋ PR コメント投稿。または [knanao/preview-cloudrun](https://github.com/knanao/preview-cloudrun) のような Action を利用。

### 案 B: GitHub Releases のみで本番デプロイ（PR ではプレビュー URL なし）

- **内容**: 本番デプロイは**タグまたは Release 作成時**にのみ実行。PR ではデプロイせず、プレビューは**ローカル `pnpm dev` のみ**。
- **メリット**: シンプルで、Cloud Run のリソースを本番 1 本に限定できる。
- **デメリット**: PR 単位の「公開 URL での確認」ができない。

### 案 C: GitHub Environments で PR 用環境の URL を登録

- **内容**: GitHub の「Environments」に PR 用環境を用意し、PR 時に Cloud Run にデプロイしたうえで、その環境の**デプロイメント URL** を API で登録する。PR の「View deployment」から開けるようにする。
- **メリット**: GitHub UI 上でデプロイ先が分かりやすい。
- **デメリット**: 結局「PR 時に Cloud Run にデプロイする」部分は案 A と同様の実装が必要。Environment の URL 更新の手間が増える。

---

## 3. 採用方針（固定）

**案 A: PR 時に Cloud Run へプレビュー用リビジョンをデプロイし、PR コメントに URL を投稿** を採用する。

- **理由**: PR ごとに確認用 URL を得られるため、レビューと「サイトを確認する方法」が明確になる。本番は main またはタグ/Release 時のデプロイに分離すれば、運用も整理しやすい。
- **実装方針**:
  - GitHub Actions で、`pull_request` の push 時に Next.js をビルド→コンテナイメージをビルド→Artifact Registry に push→Cloud Run に新リビジョンとしてデプロイ（トラフィック 0%、プレビュー用）。
  - デプロイ後に Cloud Run のリビジョン URL を取得し、GitHub API で PR にコメント投稿（または既存コメントを更新）。
  - 本番用デプロイは `push to main` または `release published` で別ジョブとし、本番用サービス・リビジョンに 100% トラフィックを流す。
- **プレビュー用リビジョン**は、必要に応じて「PR クローズ時に削除」「一定数以上は古いものから削除」などのクリーンアップを後から追加可能とする。
