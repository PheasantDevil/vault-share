# モノレポ vs 複数リポジトリ（設計前調査）

## 目的

フルスタック（フロント・API・Chrome拡張・IaC）を 1 リポジトリにまとめるか、サービスごとに分割するかを比較する。

## 定義

- **モノレポ**: 複数プロジェクト・サービスを 1 リポジトリで管理。
- **マルチリポジトリ（ポリレポ）**: サービス・プロジェクトごとに別リポジトリ。

## モノレポの利点

- **協業・コード共有**: コンポーネント・型・インフラを一括で共有しやすい。
- **原子性**: フロント・API・インフラの変更を 1 コミット・1 PR で出せる。
- **可視性・責任範囲**: 変更の影響が一覧しやすく、セキュリティレビューも行いやすい。
- **オンボーディング**: 1 回の clone とセットアップで開発を始められる。

## マルチリポの利点

- **疎結合**: サービスごとにビルド・CI/CD・デプロイを分けられる。
- **チームの独立性**: リポジトリが小さいと、担当チームが自律しやすい。
- **権限・スコープ**: リポジトリ単位でアクセス制御しやすい。

## 考慮点（Buildkite / Graphite 等の知見）

- **組織規模**: 多くのプロダクトは Google/Uber 規模の課題を持たない。過度なマルチレポ化はオンボーディング・一貫性のコストを増す。
- **ナノリポのコスト**: リポジトリを細かくしすぎると、ローカル環境のセットアップや依存の揃えが重くなる。

## 本プロジェクト向け結論

- **推奨**: **モノレポ**。
  - 少人数チーム・少人数規模のグループ共有サービスであり、フロント・API・拡張・IaC を同じタイミングで変更することが多い。
  - 型定義・API スキーマ・認証ロジックを共有するメリットが大きい。
  - CI/CD は GitHub Actions の 1 ワークフローで、変更パスに応じてビルド・デプロイを分岐する形で十分対応可能。
- **マルチレポを検討する目安**: チームが複数に分かれ、サービスが独立したライフサイクルでリリースされるようになった段階。
